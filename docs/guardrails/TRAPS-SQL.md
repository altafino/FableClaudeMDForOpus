You are here because CODE.md C7 pack dispatch fired: you are writing SQL, query-builder calls, or ORM query code. No row covers a load-bearing behavior? Probe it on the target engine and paste the output — never guess.

## NULL three-valued logic
- `col != 'x'` EXCLUDES NULL rows; `NULL = NULL` is not TRUE -> IS NULL / IS NOT DISTINCT FROM; state NULL behavior for every filter on a nullable column.
- `NOT IN (subquery)` containing one NULL matches NOTHING -> NOT EXISTS.
- COUNT(col) skips NULLs; COUNT(*) counts rows — choose deliberately and say which.

## Joins & aggregates
- LEFT JOIN + a WHERE condition on the right table becomes an INNER JOIN -> move the condition into ON.
- Joining one-to-many BEFORE SUM/COUNT double-counts -> aggregate in a subquery/CTE first; sanity-check one total against the base table.
- GROUP BY with non-aggregated selected columns: engines differ (error vs arbitrary row) -> aggregate or group every selected column.

## Indexes, types, pagination
- A function on an indexed column (`WHERE lower(email) = ...`) and leading-wildcard `LIKE '%x'` skip the index -> expression index, or restructure the predicate.
- Implicit coercion (`'123' = 123`) differs per engine and skips indexes -> match types explicitly.
- OFFSET pagination skips/duplicates rows while data changes -> keyset pagination (`WHERE id > :last ORDER BY id LIMIT n`).
- timestamp vs timestamptz: naive timestamps shift with the session timezone -> timestamptz, or document the invariant.

## Dialect & ORM
- Upsert syntax is per-engine (ON CONFLICT vs ON DUPLICATE KEY) -> read the engine + version from the project config first (CODE.md C6).
- ORM autogenerated migrations see a rename as DROP+ADD (data loss) -> review the generated migration line by line (docs/guardrails/DATA.md DA3).
