## Problem Statement

The Guardrails Kit v1.0 is complete: a portable CLAUDE.md core (routing table, 12 iron rules, 4 hard stops) plus 8 on-demand guardrail docs and MIGRATE.md, designed to make Opus/Sonnet-class models operate near Fable level inside Claude Code. It has been adversarially reviewed (193 findings applied) but **never field-tested**, and it is **prose-only**: every rule depends on the model choosing to obey it. The kit's own README names the risk — "always-on compliance is roughly constant-sum" — and under pressure the rules most likely to be dropped are exactly the ones guarding the moment of pressure. This doc maps what to improve or add to reach the kit's goal better.

## What Makes This Cool

The kit already has greppable compliance markers (`TRIGGER:`, `V1:`–`V12:`, `CAUSE:`, `ATTEMPT n`) baked into every rule. That means the enforcement and measurement layer doesn't need to be invented from scratch — the kit was built auditable, and nobody has built the auditor yet. Converting even 4–5 iron rules from "hoped-for prose" to deterministic Claude Code hooks would make the kit the first CLAUDE.md framework where the model *cannot* skip the guardrail, not merely *shouldn't*.

## Constraints

- "NO CODE CHANGES, just document" — this session's deliverable is this doc only (user, 2026-07-07).
- Kit files must never be paraphrased when carried/edited — verbatim discipline (kit's own rule, docs/STATE.md).
- Always-on core budget is hard: ≤60 kit lines, ≤15 iron rules, ≤5 CAPS lines (_FORMAT.md F3/F4). Improvements must live in hooks, tooling, or routed docs — never grow the core.
- Any kit-file edit follows _FORMAT.md F15: version bump on line 1 + README `## Upgrade notes` entry.
- Never push without explicit confirmation (user global).

## Premises

Agreed by user, 2026-07-07:
1. The kit's binding constraint is **compliance, not coverage** — the rules are good; the risk is Opus not firing them under pressure. More rule text won't close the gap; enforcement and measurement will.
2. The kit is prose-only today, but Claude Code offers deterministic mechanisms (PreToolUse/Stop/PreCompact/SessionStart hooks, native skills) that could enforce iron rules mechanically — the single biggest untapped lever.
3. v1.0 has never been field-tested; without a transcript-audit feedback loop and a before/after eval, any v1.1 tuning is guesswork.
4. The always-on budget (~45 lines) is load-bearing — improvements must not grow the core.

## Approaches Considered

### Approach A: Docs-only quick wins
Summary: fix the gaps the kit can close within its current docs-only shape. Effort: S. Risk: Low. Reuses: existing doc structure, _FORMAT.md contracts.

1. **POSIX install commands in README.** The fresh-install section is PowerShell-only (`Copy-Item`, `New-Item`) — violating the kit's own cross-platform pairing convention (MIGRATE.md gives every command as `POSIX | PowerShell`). Add the `cp`/`mkdir -p` pair.
2. **PROJECT.md template.** CLAUDE.md's `## Project` cap and MIGRATE.md M6c both route conditional content to `docs/guardrails/PROJECT.md`, but no template ships — fresh installs must invent the shape. Ship a skeleton with situation-phrased `##` anchor examples and matching zone-2 pointer-line examples.
3. **TRAPS.md language split.** Current traps are JS/Python-heavy; Go, Rust, SQL, shell are thin or absent (notable gaps: SQL parametrization, path/filename handling incl. Windows separators, character encoding, non-async concurrency/races). TRAPS.md is at 1,077 words — near its F11 **word** cap (~1,100 words; lines are fine at 60), so growth means splitting into per-language files (TRAPS-JS.md, TRAPS-PY.md, TRAPS-GO.md, TRAPS-COMMON.md) with CODE.md C7 routing by file extension — a sanctioned split ("a doc that must grow past either splits by trigger").
4. **Model-compat notes.** README says "Opus is fine" for migration; add a short tested-models matrix (Opus 4.8 / Sonnet / Haiku) with known deltas once field data exists. Placeholder until B ships numbers — this item deliberately lands with Phase B's eval results, not in Phase A's criteria.
5. **Repo housekeeping.** docs/STATE.md `## Facts` still names the authoring machine's Windows path as kit root; refresh for this checkout.

### Approach B: Enforcement + measurement layer
Summary: stop relying on model obedience for the highest-stakes rules; build the feedback loop the kit was designed for. Effort: L. Risk: Med (hook portability, transcript-format coupling). Reuses: the kit's greppable markers, Claude Code hooks API.

1. **PreToolUse hooks for iron rules** (shipped as an optional companion — `.claude/settings.json` snippet + `scripts/` dir), enforcing mechanically:
   - deny Edit/Write on a file not Read this session (iron rule 1 / CODE.md C1);
   - deny Write to an existing file (iron rule 2), pointing at the rewrite procedure;
   - deny edits under generated/vendored paths (CODE.md C2 list);
   - deny `pkill`/`taskkill /IM` by image name (hard stop 3);
   - deny `git push` unless a session allow-flag file exists (created by an explicit user-run command, e.g. `scripts/allow-push`, and cleared on session end) — hard stop 2.
   Every deny hook follows a **bypass + log convention**: a false positive (legit new-file Write, hook state lost after crash/compaction) is escaped by a documented per-hook override (env var or flag file), and every bypass is appended to a local log so the auditor (B3) can review overrides.
2. **Stop-hook verifier**: at turn end, if the reply claims done/fixed/works/passing without a `Verified:` line or V-lines in the same turn, block with a pointer to VERIFY.md. Match scope: assistant-authored prose only — quoted text, code blocks, and echoed user words are excluded to avoid false positives. Converts the echo protocol from prose to gate.
3. **Transcript compliance auditor**: a script that parses a Claude Code session transcript (JSONL), detects trigger events (non-zero exits, 3rd-file edits, done-claims, compaction returns), and checks for the expected markers — outputting a scorecard: fired / missed per rule ID. This is exactly the tuning loop STATE.md `## Next` calls for, automated.
4. **Eval harness + metrics definitions**: a fixed task set (bug-fix, refactor, multi-file feature, resume-after-compaction) run with/without the kit on Opus and Sonnet, **N runs per task per condition (N≥5)** to handle model stochasticity — report means with spread, never single runs. Metrics: marker-fire rate at trigger events, false-"done" rate, introduced-defect rate, tokens per task. Published in README so every v1.x edit is evidence-based.

### Approach C: Harness-native integration
Summary: make the routing survive the kit's most fragile moments using Claude Code's native machinery, without touching rule text. Effort: M. Risk: Med. Reuses: skills auto-discovery, routing table, MIGRATE M8 checks.

1. **SessionStart / PreCompact re-arm hook**: deterministically re-inject the post-compaction instruction (routing row 6 + "cached docs are invalid") instead of relying on the CLAUDE.md footer line surviving the model's attention after compaction — the kit's known weakest moment.
2. **Skill-ification as a second routing channel**: register thin native skills (`.claude/skills/guardrails-verify/`, `-debug/`, …) whose frontmatter `description:` carries the event verbs ("test failed", "about to commit", "claiming done"), each skill body being a pointer to its `docs/guardrails/*.md` (pointer, not restatement — preserving single-source rule F7). Harness-side semantic matching then backs up the model-side routing table. Precedence when both fire: the routing table wins (it is the owning contract); a skill auto-load is satisfied by the same doc Read, and duplicate Reads are harmless — the doc is the single source either way.
3. **Self-verifying installer**: one cross-platform script automating the fresh-install copy + the M8 hash/marker checks (items 3, 4, 6), so "installed correctly" is a command, not a claim.
4. **STATE.md freshness nudge**: a cheap statusline/hook indicator showing whether docs/STATE.md exists and when it was last updated — surfacing SESSION.md S3 drift without spending model attention.

## Recommended Approach

**Phased: A → C → B** (user-approved 2026-07-07). Rationale: A's fixes are hours of work and remove real install friction now; C makes the existing prose kit survive its most fragile moment (compaction) at moderate cost; B is the largest lever — it attacks the agreed binding constraint (compliance) and finally answers "does the kit work at all?" with numbers — and benefits from landing last, because the auditor and evals then measure the A+C improvements too.

Packaging decision to carry through all phases: hooks/scripts ship as an **optional companion layer** (e.g. `hooks/` in this repo with its own install step), keeping the core kit a pure documentation set — MIGRATE.md and the README's "portable docs" claim stay true, and users on other harnesses lose nothing.

## Open Questions

1. Companion packaging: same repo (`hooks/` + README section) or a sibling repo? Same-repo is simpler; sibling keeps the kit's "docs-only" identity strictly clean.
2. Windows hook parity: Claude Code hooks run shell commands — do the enforcement hooks need PowerShell twins, per the kit's own cross-platform principle?
3. Transcript format coupling: the auditor targets Claude Code's session JSONL — how stable is that format across CLI versions, and should the auditor pin/detect versions?
4. Eval task set: synthetic micro-tasks (reproducible, cheap) vs. real-repo tasks (representative, noisy) — likely a small mix; needs a decision before B starts.
5. Read-tracking fidelity: the "deny Edit on un-Read file" hook needs per-session Read state; confirm hooks can maintain it (state file keyed by session ID) without false positives after compaction.

## Success Criteria

- **Phase A done when**: fresh install works copy-paste on Linux/macOS (POSIX) and Windows (PowerShell); PROJECT.md template ships; TRAPS split **design** lands (file layout + C7 routing wording; the split itself executes when a language file first grows past F11 caps); all changes carry F15 version bumps + README upgrade notes.
- **Phase C done when**: after a forced compaction in a test session, the re-arm hook demonstrably fires and the model Reads SESSION.md before any file-modifying call; a skill-ified doc's description demonstrably triggers its doc Read in a test session where the routing table was not cited; the STATE.md freshness indicator visibly reflects a stale vs. fresh docs/STATE.md; installer script reproduces MIGRATE M8 items 3/4/6 with zero manual steps.
- **Phase B done when**: each PreToolUse deny hook blocks its trigger case AND passes its bypass+log path in a scripted test; the Stop-hook verifier blocks an unverified done-claim and passes a `Verified:`-bearing turn in the same test suite; the auditor produces a per-rule fired/missed scorecard from any session transcript in under a minute; the eval harness publishes with-kit vs. without-kit numbers for Opus on the fixed task set (provisional target: >80% marker-fire rate at trigger events — how trigger events are ground-truthed in a transcript is defined by the auditor spec, so the threshold firms up there; plus a measurable reduction in false-"done" claims vs. baseline).

## Distribution Plan

Unchanged for the kit itself: git repo copy per README (now cross-platform). Companion hooks distribute as an in-repo `hooks/` directory with a documented `.claude/settings.json` merge snippet and the self-verifying installer; no package manager needed. CI (GitHub Actions) added when B lands, to run the eval harness on kit changes — deferred until then.
